# 🧩 3.2.2 Corrupção de Índices
**Categoria:** Dados  
**CI afetado:** Tabela / Índice  
**Escopo:** Integridade Lógica / Estrutural  
**Severidade sugerida:** P2  

---

## 🎯 Objetivo

Identificar corrupção isolada em índices e determinar a estratégia adequada de reconstrução ou recriação, minimizando impacto nos dados.

---

## 🔎 Diagnóstico Inicial (Quick Triage)

```sql
DBCC CHECKDB ('NomeDoBanco') WITH NO_INFOMSGS;
```
---

🧠 **Análise Detalhada**
- **Identificar Índice Afetado**
```sql
DBCC CHECKDB ('NomeDoBanco') WITH TABLERESULTS, NO_INFOMSGS;
```
Objetivo:

- Identificar objeto afetado (object_id, index_id)
- Confirmar se corrupção está limitada ao índice

##

- **Verificar Integridade Específica**
```sql
DBCC CHECKTABLE ('NomeDaTabela') WITH NO_INFOMSGS, ALL_ERRORMSGS;
```
Objetivo:

- Confirmar corrupção localizada

---

🧩 **Possíveis Causas**

- Falha de I/O
- Problemas em storage
- Snapshot inconsistente
- Corrupção herdada de backup
- Interrupção abrupta durante rebuild

---

🛠️ **Ações de Mitigação**
1️⃣ **Rebuild do Índice (Recomendado)**
```sql
ALTER INDEX NomeDoIndice
ON NomeDaTabela
REBUILD;
```
##

2️⃣ **Recriar Índice (Quando necessário)**
```sql
DROP INDEX NomeDoIndice ON NomeDaTabela;
```
```sql
CREATE INDEX NomeDoIndice
ON NomeDaTabela (Coluna1, Coluna2);
```
##

3️⃣ **Caso Persistam Erros**

- Executar DBCC CHECKDB completo
- Avaliar restauração de backup

---

🔁 **Monitoramento Pós-Mitigation**
```sql
DBCC CHECKTABLE ('NomeDaTabela');
```
- Validar ausência de erros
- Monitorar Error Log
- Revisar rotina de manutenção de índices

---

📌 **Observação Operacional**

- Se a corrupção estiver restrita a índice não clusterizado, o rebuild normalmente resolve sem perda de dados.

- Se envolver índice clusterizado ou heap, tratar como possível corrupção estrutural do banco.
