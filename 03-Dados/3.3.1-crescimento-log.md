# 📈 3.3.1 Crescimento Anormal do Transaction Log
**Categoria:** Dados  
**CI afetado:** Arquivo LDF  
**Escopo:** Transaction Log / Recuperação  
**Severidade sugerida:** P1 / P2  

---

## 🎯 Objetivo

Identificar crescimento excessivo do arquivo de Transaction Log (LDF), analisando falhas de truncamento, transações longas ou ausência de backup de log.

---

## 🔎 Diagnóstico Inicial (Quick Triage)

```sql
DBCC SQLPERF(LOGSPACE);
```
---

🧠 **Análise Detalhada**
- **Verificar Uso do Log**
```sql
SELECT 
    name,
    log_reuse_wait_desc
FROM sys.databases;
```
Objetivo:

- Identificar motivo do log não estar sendo truncado

##

- **Sessões com Transações Ativas**
```sql
DBCC OPENTRAN ('NomeDoBanco');
```
Objetivo:

- Detectar transação aberta impedindo truncamento

##

- **Espaço Utilizado no Log**
```sql
SELECT *
FROM sys.dm_db_log_space_usage;
```
Objetivo:

- Validar consumo atual do LDF

---

🧩 **Possíveis Causas**

- Modelo FULL sem backup de log
- Transação longa aberta
- Replicação pendente
- Always On com atraso
- Job de backup falhando

---

🛠️ **Ações de Mitigação**

1️⃣ **Executar Backup de Log (Modelo FULL)**
```sql
BACKUP LOG NomeDoBanco
TO DISK = 'caminho_backup.trn';
```
##

2️⃣ **Identificar e Finalizar Transação Longa**

- Validar sessão responsável
- Avaliar impacto antes de finalizar

##

3️⃣ **Ajustar Configuração de Autogrowth**
```sql
ALTER DATABASE NomeDoBanco
MODIFY FILE (
    NAME = NomeLog,
    FILEGROWTH = 512MB
);
```
---

🔁 **Monitoramento Pós-Mitigation**
```sql
DBCC SQLPERF(LOGSPACE);
```
- Confirmar redução do uso
- Monitorar log_reuse_wait_desc
- Validar rotina de backup de log

---

📌 **Observação Operacional**

- Evitar SHRINK como solução recorrente.
- Crescimento de log é sintoma, não causa.
- Ajustar rotina de backup e analisar comportamento da aplicação.
